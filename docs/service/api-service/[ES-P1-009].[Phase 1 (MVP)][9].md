### ES-P1-009 - API Service BFF Orchestration Entrypoint for MVP Web Flows
- Task Description:
	- Implement `api-service` BFF orchestration entrypoint for MVP web flows across auth, jam, playback, and catalog.
	- Define request routing and response aggregation behavior so web client calls can use one BFF-facing API surface.
	- Standardize upstream error mapping and timeout handling for auth, jam, playback, and catalog dependencies.
- Dependencies:
	- Stable downstream contracts from `auth-service`, `jam-service`, `playback-service`, and `catalog-service`.
	- API gateway routing and identity propagation into `api-service`.
	- Shared response envelope and error contract alignment across services.
- Definition of Done:
	- `api-service` provides a working BFF entrypoint for MVP web client integration.
	- MVP web flows (auth, jam, playback, catalog) can be invoked through the BFF path with deterministic behavior.
	- Integration tests verify routing, aggregation, and failure mapping for core MVP paths.

### Test Guide
- Automated validation:
	- Run `cd backend/api-service && go test ./...`
	- Focus orchestration success: `go test ./internal/bff -run TestOrchestrationSuccessAcrossDependencies -count=1`
	- Focus timeout normalization: `go test ./internal/bff -run TestOrchestrationTimeoutNormalizedForRequiredDependency -count=1`
	- Focus partial-result behavior: `go test ./internal/bff -run TestOrchestrationOptionalFailureReturnsPartial -count=1`
	- Focus claim/catalog contract checks:
		- `go test ./internal/bff -run TestHTTPAuthClientValidateBearerToken -count=1`
		- `go test ./internal/bff -run TestHTTPCatalogClientLookupTrack -count=1`

- Manual endpoint test with curl (requires auth/jam/playback/catalog services reachable by api-service URLs):
	- Start api-service: `cd backend/api-service && go run ./cmd/server`
	- Call orchestration:
		- `curl -i -X POST "http://localhost:8084/v1/bff/mvp/sessions/jam_1/orchestration" -H "Authorization: Bearer <token>" -H "Content-Type: application/json" -d "{\"trackId\":\"trk_1\",\"playbackCommand\":{\"command\":\"pause\",\"clientEventId\":\"evt_1\",\"expectedQueueVersion\":1}}"`

- Swagger-based manual testing:
	- Open UI: `http://localhost:8084/swagger`
	- OpenAPI JSON: `http://localhost:8084/swagger/openapi.json`
	- In Swagger UI:
		- Click `Authorize` and provide bearer token value.
		- Expand `POST /v1/bff/mvp/sessions/{sessionId}/orchestration`.
		- Set `sessionId`, edit request payload, then click `Execute`.

- Expected outcomes:
	- Full success: `200` with `success=true`, `data.partial=false`, and dependency statuses marked `ok`.
	- Optional dependency degradation: `200` with `success=true`, `data.partial=true`, and one or more `data.issues` entries.
	- Required dependency timeout/unavailable: `503` with deterministic `error.code` (`dependency_timeout` or `dependency_unavailable`).
